# QA/TC 문서 자동화 시스템 작업지시서

## 1. 문서 목적

이 문서는 AI agent가 `QA/TC 문서 자동화 시스템`을 구현할 때 따라야 할 프로젝트 전용 작업지시서다.  
공통 작업 원칙은 기존 `AGENTS.md`를 따르고, 이 문서는 해당 프로젝트의 목표, 범위, 우선순위, 금지사항, 완료 조건을 정의한다.

---

## 2. 프로젝트 목표

요구사항서, 화면설계서, API 명세서, Excel 문서를 기반으로  
테스트케이스(TC) 초안과 RTM(요구사항 추적 매트릭스)을 자동 생성하는  
사내 전용 AI 보조 시스템을 구축한다.

이 시스템의 목적은 QA 판단을 대체하는 것이 아니라,  
문서 작성 공수를 줄이고 누락을 줄이며 추적성을 높이는 것이다.

---

## 3. 시스템 목표 범위

### 포함 범위
- PDF, Word, Excel 문서 파싱
- 요구사항 ID 기반 문서 분할 및 인덱싱
- RAG 기반 문맥 검색
- JSON 스키마 기반 테스트케이스 초안 생성
- 테스트케이스 검증 로직 수행
- Excel 템플릿 기반 결과 출력
- RTM 자동 생성
- 사용자 검토 및 승인 UI 제공

### 제외 범위
- QA 최종 판단 자동화
- 테스트 실행 자동화
- 외부 클라우드 API 기반 추론
- 근거 없는 요구사항 추론 및 임의 생성

---

## 4. 핵심 원칙

- 반드시 폐쇄망 또는 로컬 환경에서 동작 가능한 구조로 구현한다.
- 자유서술형 출력보다 구조화된 JSON 출력을 우선한다.
- 문서 근거가 없는 내용은 임의로 생성하지 않는다.
- 불확실한 내용은 억지로 채우지 말고 `확인 필요` 상태로 분리한다.
- 승인되지 않은 결과는 최종 산출물로 취급하지 않는다.
- 모든 결과는 요구사항 ID 기준으로 추적 가능해야 한다.
- 문서 버전, 모델 버전, 프롬프트 버전이 추적 가능해야 한다.

## 7. 기술 방향

### 7.1 로컬 AI 인프라
- 모델 서빙은 로컬 또는 온프레미스 환경을 전제로 한다.
- 추론 서버는 `vLLM` 기반 API 구조를 우선 검토한다.
- 모델은 한국어 이해, 구조화 출력 안정성, 양자화 안정성, 온프렘 배포 용이성을 기준으로 선정한다.
- 특정 모델명에 하드코딩하지 말고 교체 가능 구조로 구현한다.

### 7.2 문서 파싱
- PDF는 `PyMuPDF` 계열을 우선 검토한다.
- Word는 `python-docx` 계열을 사용한다.
- Excel은 `openpyxl` 기반으로 처리한다.
- 표 구조가 무너지지 않도록 파싱 품질을 우선 확인한다.
- OCR은 꼭 필요한 경우에만 제한적으로 사용한다.

### 7.3 저장 및 검색
- 문서 청킹은 글자 수 기준이 아니라 아래 기준을 우선 적용한다.
  - 요구사항 ID 단위
  - 기능 단위
  - 화면 단위
  - API 단위
  - 예외 규칙 단위
- 검색은 벡터 검색만 사용하지 말고, 키워드 검색과 결합한 하이브리드 검색 구조를 우선한다.
- 초기 구현은 단순 구조로 가되, 나중에 확장 가능한 형태로 설계한다.
- 문서 변경 시 전체 재처리만 하지 말고 증분 인덱싱이 가능하도록 고려한다.

### 7.4 테스트케이스 생성
- 출력은 반드시 JSON 스키마를 따르도록 강제한다.
- 테스트케이스는 요구사항 ID와 연결되어야 한다.
- 숫자, 날짜, 길이 제한이 있는 경우 경계값 분석(BVA)을 포함해야 한다.
- 정상, 비정상, 예외 흐름을 구분해야 한다.
- 기대결과는 구체적으로 작성해야 한다.
- 원문에 없는 요구사항을 상상해서 생성하지 않는다.

### 7.5 문서 출력 자동화
- 생성 결과는 `Pandas` 기반 정리 후 Excel 템플릿에 반영한다.
- 원본 템플릿 파일은 직접 수정하지 않는다.
- 반드시 복사본에만 출력한다.
- RTM 시트를 자동 생성해야 한다.
- 요구사항 ID와 TC ID의 연결 상태를 확인할 수 있어야 한다.
- 누락된 요구사항은 시각적으로 표시할 수 있어야 한다.

### 7.6 사용자 검토 및 승인
- UI는 초기에 Streamlit 기반으로 구현 가능하다.
- 운영 단계에서는 FastAPI + 별도 UI 구조로 확장 가능해야 한다.
- 결과 상태는 최소 아래를 지원해야 한다.
  - 초안
  - 검토중
  - 수정완료
  - 승인
  - 반려
- 사용자의 수정 이력은 저장해야 한다.
- AI 초안과 사용자 수정본의 차이를 비교 가능해야 한다.

## 8. 테스트케이스 JSON 표준 예시

아래 필드는 기본적으로 포함되도록 구현한다.

- `tc_id`
- `requirement_id`
- `feature_name`
- `preconditions`
- `test_steps`
- `test_data`
- `expected_result`
- `test_type`
- `priority`
- `source_chunks`
- `review_status`

필드명은 내부 표준으로 관리하고, 출력 템플릿에 따라 매핑 가능하도록 구현한다.

---

## 9. 검증 규칙

생성 결과는 아래 검증을 통과해야 한다.

### 필수 검증
- JSON 파싱 가능 여부
- 필수 필드 존재 여부
- 필드 타입 유효성
- requirement_id 존재 여부
- test_steps 비어 있지 않음
- expected_result 비어 있지 않음

### 품질 검증
- 중복 테스트케이스 탐지
- 요구사항 대비 테스트케이스 누락 여부 확인
- 숫자/날짜/길이 제한 필드의 BVA 포함 여부
- 인증/권한 기능의 실패 케이스 포함 여부
- 예외 흐름 포함 여부
- 요구사항 근거 없는 추론 여부 점검

### 실패 처리 원칙
- 검증 실패 결과는 최종 산출물로 내보내지 않는다.
- 재생성 또는 사용자 검토 대기 상태로 전환한다.
- 실패 원인을 로그에 남긴다.

---

## 10. RTM 생성 규칙

- RTM은 요구사항 ID 기준으로 생성한다.
- 각 요구사항에 연결된 TC ID 목록을 표시한다.
- 요구사항 누락 여부를 자동 검출한다.
- 중복 연결 여부를 확인한다.
- 가능하면 생성 근거 문서 chunk 정보도 함께 연결한다.

## 16. 완료 조건

아래 조건을 만족하면 1차 구현 완료로 본다.

- 문서 업로드가 가능하다.
- 요구사항 ID 기준으로 문서를 검색할 수 있다.
- 테스트케이스를 유효한 JSON으로 생성할 수 있다.
- 검증 통과 여부를 사용자에게 보여줄 수 있다.
- RTM이 자동 생성된다.
- 승인된 결과를 Excel로 다운로드할 수 있다.
- 생성 근거와 요구사항 ID를 추적할 수 있다.

---

## 17. AI agent 작업 방식

- 구현은 작은 단위로 나누어 진행한다.
- 한 번에 전체를 만들기보다, 파서 → 검색 → 생성 → 검증 → 출력 → UI 순서로 나눈다.
- 각 단계마다 실행 가능한 결과를 남긴다.
- 코드 작성 후에는 반드시 실패 케이스를 먼저 점검한다.
- 복잡성을 불필요하게 키우지 말고, MVP 범위에서 먼저 완성한다.
- 추후 확장 가능성을 고려하되, 초기 구현에 과도한 기능을 넣지 않는다.
